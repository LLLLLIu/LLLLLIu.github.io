什么是版本控制系统？

版本控制系统（即VCS）会跟踪人员和团队在一起合作项目时的变化历史。随着项目的发展，团队可以随时运行测试，修复错误并贡献新代码，并确信可以恢复任何版本。开发人员可以查看项目历史记录以找出：

  • 做了哪些更改？
  • 谁做出了改变？
  • 什么时候做出改变？
  • 为什么需要更改？

什么是分布式版本控制系统？

Git是通常用于开源和商业软件开发的分布式版本控制系统（DVCS）的一个例子。DVCS允许完全访问项目的每个文件，分支和迭代，并允许每个用户访问所有更改的完整和独立的历史记录。与一度流行的集中式版本控制系统不同，DVS如Git不需要连接到中央存储库。开发人员可以在任何地方工作，并可以从任何时区进行异步协作。

如果没有版本控制，团队成员将面临冗余任务，较慢的时间表和单个项目的多个副本。为了消除不必要的工作，Git和其他VCS为每个贡献者提供了对项目的统一和一致的视图，从而显示出已经在进行中的工作。看到透明的变更历史，由谁制作，以及他们如何为项目的开发做出贡献，可以帮助团队成员在独立工作时保持一致。

为什么选择Git？

根据最新的Stack Overflow开发者调查，超过70％的开发者使用Git，使其成为全球使用最多的VCS。Git通常用于开源和商业软件开发，对个人，团队和企业都有很大的好处。

  • Git让开发人员可以在一个地方看到他们的变化，决策和任何项目进程的整个时间表。从他们访问项目历史的那一刻起，开发人员就有了理解它并开始贡献的所有环境。
  • 开发人员在每个时区工作。使用像Git这样的DVCS，可以在任何时候进行协作，同时保持源代码的完整性。使用分支机构，开发人员可以安全地提出更改生产代码
  • 使用Git的企业可以打破团队之间的沟通障碍，并让他们专注于做最好的工作。另外，Git可以让整个企业的专家与主要项目进行协作。

什么是资料库？

一个仓库，或Git项目，包括与项目相关的文件和文件夹的整个集合，每个文件的修订历史记录一起。文件历史记录在时间上称为提交，并且提交以链接列表关系的形式存在，并且可以组织为多行开发（称为分支）。由于Git是一个DVCS，存储库是自包含的单元，任何拥有存储库副本的人都可以访问整个代码库及其历史记录。使用命令行或其他易用的界面，git存储库还允许：与历史记录交互，克隆，创建分支，提交，合并，比较不同版本的代码中的更改等等。

在存储库中工作可以保持开发项目的组织和保护。我们鼓励开发人员修复bug或创建新功能，而不用担心干扰主线开发工作。Git通过使用主题分支来促进这一点：轻量级指向历史提交的指针，当不再需要时可轻松创建和弃用。

通过GitHub等平台，Git还为项目透明度和协作提供了更多机会。公共仓库可以帮助团队一起工作，以建立尽可能最好的最终产品。

基本的Git命令

要使用Git，开发人员使用特定的命令来复制，创建，更改和组合代码。这些命令可以直接从命令行或使用GitHub Desktop或Git Kraken 等应用程序执行。以下是使用Git的一些常用命令：

  • git init初始化一个全新的Git存储库并开始跟踪一个现有的目录。它在现有目录中添加一个隐藏的子文件夹，该目录包含版本控制所需的内部数据结构。
  • git clone创建一个远程存在的项目的本地副本。克隆包括所有项目的文件，历史和分支。
  • git add阶段变化。Git会跟踪开发者代码库的变化，但有必要对这些变化进行分类并拍摄快照，以便将其纳入项目历史记录中。此命令执行分段，这是两步过程的第一部分。所有将进行的更改都将成为下一个快照的一部分，并成为项目历史的一部分。分别进行分段和提交可以让开发人员完全控制其项目历史，而不必改变他们的代码和工作方式。
  • git commit将快照保存到项目历史记录并完成变更跟踪过程。简而言之，提交功能就像拍照一样。任何已经上演的内容git add都将成为快照的一部分git commit。
  • git status 显示更改的状态为untracked，modified或staged。
  • git branch 显示当地工作的分行。
  • git merge合并发展线。此命令通常用于组合在两个不同分支上所做的更改。例如，当开发人员想要将功能分支的更改组合到主分支中进行部署时，他们会合并。
  • git pull使用来自远程对应方的更新更新本地开发线。如果队友向远程分支提交提交，开发人员使用此命令，他们想要在其本地环境中反映这些更改。
  • git push 使用本地提交到分支的任何提交来更新远程存储库。

从Git命令的完整参考指南中了解更多信息。

GitHub如何适应

GitHub是一个Git托管资源库，它为开发人员提供了通过命令行功能，问题（线程讨论），拉取请求，代码审查或在GitHub市场中使用一系列免费和应用购买应用程序来发布更好的代码的工具。借助GitHub流程等协作层，一个由1500万开发人员组成的社区，以及拥有数百个集成的ecosystm，GitHub改变了软件构建的方式。

GitHub如何工作

GitHub将协作直接构建到开发过程中。工作被组织到仓库中，devlopers可以概述需求或方向，并为团队成员设定期望值。然后，使用GitHub流程，开发人员只需创建一个分支来处理更新，提交更改以保存更新，打开拉取请求来提出并讨论更改，并在每个人都在同一页面上时合并拉取请求。

GitHub流程

GitHub流程是一个轻量级的，基于分支的工作流程，围绕全球团队（包括我们的团队）使用的核心Git命令构建。

GitHub流程有六个步骤，每个步骤在实施时都有不同的好处：

  1. 创建分支：从规范部署分支（通常master）创建的主题分支允许团队为许多并行工作做出贡献。短命的主题分支尤其能够保持团队的专注，并使结果更加快速。
  2. 添加提交：分支内开发工作的快照在项目历史记录中创建安全，可恢复的点。
  3. 打开拉取请求：拉取请求宣传一个项目正在进行的努力，并为透明的开发过程确定基调。
  4. 讨论和评审代码：团队通过评论，测试和审查公开拉取请求参与代码评审。代码审查是开放和参与式文化的核心。
  5. 合并：单击合并后，GitHub会自动执行相应的本地“混帐合并”操作。GitHub还将整个分支开发历史记录保存在合并的pull请求中。
  6. 部署：团队可以选择最佳的发布周期，或者合并持续集成工具，并确保部署分支上的代码已经通过了一个强大的工作流程。

协作开发模型

人们在GitHub上进行协作有两种主要方式：

  1. 共享存储库
  2. fork和pull

通过共享存储库，个人和团队被明确指定为具有读取，写入或管理员访问权的贡献者。这种简单的权限结构与受保护的分支机构和Marketplace等功能相结合，可以帮助团队在采用GitHub时快速进展。

对于开源项目或任何人都可以贡献的项目，管理个人权限可能具有挑战性，但分叉模式允许任何能够查看项目贡献的人。分叉是开发者个人帐户下项目的副本。每个开发人员都可以完全控制自己的叉子，并且可以自由地实施修复或新功能。用叉子完成的工作要么保持分开，要么通过拉动请求回到原来的项目。在那里，维护人员可以在合并之前查看建议的更改。有关更多信息，请参阅分叉项目指南。
